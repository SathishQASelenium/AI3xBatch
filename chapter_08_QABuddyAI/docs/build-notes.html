Request lifecycle

Flask holds one process-wide Store — the embedded Qdrant client only allows one owner of qdrant_data/, so chat and ingest share it rather than opening their own.

01

POST /api/chat opens an SSE stream

Flask wraps the generator in stream_with_context; every yielded dict is serialized as one data: {...}\n\n event.

02

ask_events() drives the pipeline

condense → mode detect → retrieve → citations → relevance gate → prompt build → LLM stream. Each stage yields its own event type: meta, citations, token, done.

03

GET /api/ingest runs on a background thread

A queue.Queue bridges a daemon thread doing the actual embedding to a generator that drains the queue and re-emits each item as SSE, so the UI's progress bar updates live without blocking the request.

04

GET /api/search is the debug escape hatch

Runs retrieval only — hybrid search, RRF, rerank — with no LLM call, so retrieval quality can be checked independent of the model.

Ingestion internals

10 source folders, one shared contract: every loader returns a list of Doc(uid, text, payload).

The uid is a truncated SHA-256 of source_type|path|text — deterministic, so re-ingesting the same content always produces the same Qdrant point id (a uuid5 of that hash) and upserts overwrite in place instead of duplicating.

app/ingestion/base.pystable id
h = hashlib.sha256(f"{source_type}|{path}|{text}".encode())
uid = h.hexdigest()[:40]
point_id = uuid.uuid5(uuid.NAMESPACE_URL, uid)   # idempotent upsert

Re-ingesting is a diff, not a rebuild. Each source keeps a per-file signature in data/.ingest_manifest.json — a hash of that file's chunk uids. Unchanged files are skipped entirely; changed files are deleted and re-embedded; files removed from disk are deleted from Qdrant. The manifest only commits after a source's full loop finishes, so an interrupted run re-does that source's changed files rather than resuming mid-file.

The code loader (sources 01/02) is the most involved: it walks Java/TypeScript files line by line, tracks brace depth, and matches method/class signatures with regex (a Java-specific pattern, five TypeScript patterns for class/function/arrow/test-block/method). A unit closes when depth returns to its opening level. Oversized classes decompose into their own methods; leftover lines — imports, fields — become windowed fallback chunks; anything under 250 characters merges into its neighbor. Every chunk keeps 1-indexed line_start/line_end, which is what turns into a clickable path:49-70 citation later.

Retrieval internals

app/retrieval.py — condense, rewrite, search twice, fuse, rerank, gate.

01

Mode detection

A follow-up question is condensed against the last 3 turns via one small LLM call, then matched against three regexes before falling back to plain answer mode:

generate → create/write/draft test cases rca → root cause / why did it fail review → coverage gaps / missing
02

Query rewrite

The LLM produces 3 alternate phrasings of the question (temperature 0.4). Original + rewrites all get embedded and searched — cheap recall insurance against a single bad phrasing.

03

Hybrid search, per variant

Each query variant runs a dense_search and a sparse_search against Qdrant (top 20 each, optionally filtered by selected source types), deduped into two result maps keyed by point id.

04

Reciprocal Rank Fusion

Dense and sparse rankings merge without needing comparable score scales:

score[id] += 1 / (k + rank + 1)   # k = 60, once per list
05

Cross-encoder rerank

The fused top-12 go through bge-reranker-v2-m3 as (question, chunk_text) pairs; logits pass through sigmoid for a 0–1 score, top-6 survive.

06

Relevance gate

If the best rerank score is below 0.22, QABuddy returns a fixed "not in the knowledge base" message instead of calling the LLM at all — the anti-hallucination floor lives in retrieval, not just in the prompt.

Prompting rules

app/prompts.py — one base rule set, four mode overlays, a glossary injection.

Every mode shares the same non-negotiable base: answer only from the retrieved chunks, cite every claim as [n], use exact identifiers (file paths, line numbers, ticket keys, test-case ids) rather than paraphrasing them, and say what's missing instead of guessing. On top of that, each mode gets its own system-prompt overlay — generate asks for Title/Preconditions/Steps/Expected Result/Priority/Tags; review asks for covered areas, gaps, and risky assumptions; rca asks for root cause, evidence chain, and bug classification.

Citations are built per source type so they read naturally inside the answer — chunk_label() turns a payload into repo/path:line_start-line_end for code, test case TC-014 for a row, JIRA VWO-123 for a ticket, Jenkins build #128 for a log — the same labels the LLM sees in its context and the UI shows in the citation cards.

Bugs hit & fixed

Four real problems, in the order they surfaced.

FlagEmbedding's reranker crashed on load library

FlagEmbedding 1.4.0's FlagReranker.compute_score calls tokenizer.prepare_for_model, which the loaded XLM-R tokenizer for bge-reranker-v2-m3 doesn't expose in that version.

Fix — bypassed the wrapper entirely: load the model as a plain AutoModelForSequenceClassification via transformers, tokenize the (query, chunk) pairs directly, and sigmoid the logits ourselves. Version-robust, one dependency fewer in the hot path.

Streamed answers mangled quotes and dashes encoding

Groq's SSE response carries no explicit charset on its content-type, so requests silently falls back to latin-1 when iterating lines — curly quotes and em-dashes came through corrupted mid-stream.

Fix — decode every raw line as UTF-8 ourselves before parsing the data: ... payload, instead of trusting requests' auto-detection.

Killing ingest mid-file forced a full re-embed ingestion

The manifest signs each source per file, and only writes after that source's whole loop completes. Source 03 (the 5,000-row test-case CSV) is one file — an interrupted run left no partial credit, so the next run re-embedded all 5,000 rows from row one.

Fix — not a code change here — trimmed the working CSV to 200 rows for a fast local demo, kept the full 5,000-row file as a backup to swap back in for a real ingest run.

dev.sh assumed a Linux venv layout platform

The convenience script calls .venv/bin/python directly. On Windows, uv venv creates .venv/Scripts/python.exe — the script fails with "No such file or directory" before Flask ever starts.

Fix — ran the server module directly with the Windows interpreter path (.venv/Scripts/python.exe -m app.server.app) instead of the shell wrapper.
